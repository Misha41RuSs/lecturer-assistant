# Введение для команды
Данный документ — это ваша дорожная карта по обеспечению качества (QA) в микросервисном приложении. Архитектура разбита на независимые узлы, поэтому и тестирование тут разбито так, чтобы каждый из вас отвечал за конкретный слой логики и никто не мешал друг другу. 

---

### QA Engineer 1 (Automation Backend Core)
**Зона ответственности:** Базовая инфраструктура тестирования, `content-service` (всё, что связано с файлами и их обработкой), `admin-gateway` (входная точка).

- **Задачи и что они значат на практике:**
  - **Настройка тестовой среды:**
    - Изоляция среды: использование *Testcontainers* для автоматического поднятия PostgreSQL прямо в Docker при старте тестов. Тесты не должны зависеть от БД, установленных на ваших локальных машинах. Для тестирования файловой логики нужно настроить S3 Mock (например, MinIO).
  - **Unit-тесты для парсеров (Сердце content-service):**
    - *Покрытие GIFT:* Это самый приоритетный парсер. Нужно подготовить набор `.txt` файлов: идеальные (со всеми форматами вопросов) и намеренно "битые" (например, нет правильного ответа, сломаны скобки). Парсер не должен возвращать 500-ю ошибку — он должен выбрасывать понятные кастомные Exception, например `ValidationException`.
    - *PDF/PPTX:* Для smoke-тестов достаточно проверить, что нормальные файлы распиливаются на картинки. Для негативных кейсов — подсунуть `.exe` с расширением `.pdf` или запароленный документ.
  - **Интеграционные тесты `content-service`:**
    - Подняв реальную базу через Testcontainers, сымитировать HTTP-запрос контроллера и убедиться, что лекция не просто создается в памяти, но и корректно коммитится в таблицу БД со всеми связями.
  - **Тестирование `admin-gateway`:**
    - Проверка Edge-кейсов роутинга: пропускает ли Gateway нужные запросы и отбрасывает ли запросы с огромными заголовками или без необходимых прав.

- **Главный приоритет:**
  Надёжность загрузки контента. Приложение должно следовать принципу *fail-fast* — если загружен невалидный файл, ошибка должна отстреливать сразу в Gateway или `content-service`, не укладывая микросервис.

### QA Engineer 2 (Service Backend & Bot Logic)
**Зона ответственности:** Бизнес-логика основного учебного процесса (`lecture-broadcasting-service`, `quiz-service`, `analytics-service`), Telegram-бот.

- **Задачи и что они значат на практике:**
  - **Unit и Integration тесты бизнес-логики:**
    - *Жизненный цикл лекции:* Проверить конечный автомат состояний лекции (Draft → Active → Closed). Нельзя активировать уже закрытую лекцию, нельзя закрыть драфт напрямую.
    - *Логика оценивания:* В `quiz-service` написать Mockito-тесты, эмулирующие разные правильные и неправильные ответы, частичные совпадения (если есть множественный выбор) и проверку итогового score.
  - **Тестирование Telegram-бота (Без интернета!):**
    - *Мокирование webhook:* Мы не будем посылать настоящие сообщения в реальный API Телеграма во время автотестов. Мы формируем локальные JSON-объекты (как будто они пришли от серверов Telegram) и отправляем их напрямую в Webhook-контроллер бота.
    - *Изоляция внешних вызовов:* Исходящие RestTemplate/Feign/WebClient вызовы мокаем через Mockito, чтобы проверять только то, как бот реагирует на текст (проверка ответов, регистрация студента).
  - **Тестирование конкурентности (базовый уровень):**
    - Если два потока (или два запроса) одновременно пытаются ответить на один тест для одного студента — проверить, что запишется только один результат (Идемпотентность).

- **Главный приоритет:**
  Чтобы процесс проведения пары через Telegram работал безупречно, а баллы считались со 100% точностью.

### QA Engineer 3 (System Integration & Performance SDET)
**Зона ответственности:** Сквозные (E2E) API-тесты всей экосистемы сразу, проверка производительности (нагрузка), контракты между микросервисами.

- **Задачи и что они значат на практике:**
  - **Разработка System Integration Framework:**
    - Использование REST Assured для HTTP-вызовов всех микросервисов в связке.
    - Использование *Awaitility*: так как парсинг и многие системы асинхронны (работают через очереди или Background-генерацию), нельзя использовать `Thread.sleep(5000)`. Нужно настроить Awaitility: "пинговать статус лекции каждые 500мс до тех пор, пока статус не станет READY".
    - Максимальный сценарий: Закинуть лекцию -> дождаться загрузки -> запустить пару -> "переобуть" бота -> дернуть webhook от 5 студентов. Всё только через код.
  - **Контрактное тестирование (Pact / Spring Cloud Contract):**
    - Гарантия того, что если `content-service` внезапно переименует поле `score` в `points` в своем JSON, то билд сразу упадет, потому что контракт с `quiz-service` будет нарушен.
  - **Нагрузочное тестирование (Gatling):**
    - Использовать Gatling на Scala/Java. Цель: сгенерировать фейковый трафик, имитируя 100 студентов, которые ОДНОВРЕМЕННО жмут кнопку ответа в Telegram в одну секунду.
    - Смотрим на метрики: не вырос ли Response Time > 2 секунд? Не заблокировалась ли БД от обилия апдейтов статистики в `analytics-service`?
  - **Безопасность (Security IDOR):**
    - IDOR (Insecure Direct Object Reference) — проверить, что Студент А, случайно или намеренно отправив HTTP-запрос с ID лекции Студента Б, получит `403 Forbidden`, а не доступ к чужой лекции.

- **Главный приоритет:**
  Система не должна просто "работать" для одного пользователя. Она должна выживать под реальной студенческой толпой и быть монолитным механизмом, который невозможно обрушить.

### Общие рекомендации по архитектуре тестирования 
*(Для всей команды, чтобы не ссориться!)*

- **Разделяем уровни тестирования:** Не пытайтесь проверять математику в квизах через E2E (QA 3), это долго. Математика проверяется Unit-тестами за доли секунд (QA 2). E2E проверяет только, что сервисы могут общаться.
- **Никаких внешних зависимостей:** Использование Testcontainers для БД — закон. Удалили контейнер -> развернули новый. Чистая среда каждый раз.
- **Никакого реального Телеграма:** Telegram API не тестируется напрямую — только через webhook. Не надо завязываться на интернет.
- **Скажем НЕТ `Thread.sleep`:** Минимизировать использование sleep → использовать polling (Awaitility). Если использовать sleep, ваши E2E тесты рано или поздно начнут идти 30+ минут и все перестанут их запускать.
- **Телеметрия:** Логирование, сбор CPU и метрики JVM обязательны для QA 3 при анализе нагрузки, иначе вы не поймете, почему сервис лег (сдохла память или лег коннект к БД?).
